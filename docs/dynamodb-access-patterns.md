# Technical Specification: DynamoDB Access Patterns via AWS Amplify Gen2

**Target Audience:** Architect / LLM Agent  
**Application:** TeamTrack — Soccer Game Management PWA  
**Stack:** React + TypeScript, AWS Amplify Gen2, AppSync (GraphQL), DynamoDB

---

## 1. Overview

TeamTrack accesses DynamoDB exclusively through AWS AppSync GraphQL, generated by Amplify Gen2. There is no direct DynamoDB SDK access from the frontend. All reads and writes flow through one of three mechanisms:

| Mechanism | When Used | Consistency |
|---|---|---|
| `observeQuery` | Real-time subscriptions for live UI state | Eventual |
| `.list()` | One-time paginated reads, fallback queries | Eventual |
| `.get()` | Single-item lookup by primary key | Eventual |
| Custom Lambda or custom AppSync resolver | Server-authoritative reads or write workflows | Can use strong reads internally |

AppSync resolves all queries against DynamoDB using **eventually consistent reads by default**. Strong consistency is available at the DynamoDB level but **cannot be requested through the Amplify Gen2 GraphQL client**. AppSync resolvers control consistency internally. To use strong consistency, a custom Lambda resolver with direct DynamoDB SDK access is required.

## 2. Current Access Pattern Inventory

### 2.1 `observeQuery` — Real-Time Subscriptions

Used throughout the application via the `useAmplifyQuery` hook and directly in `useGameSubscriptions` and `useTeamData`. This is the dominant read pattern.

**How it works:**  
`observeQuery` combines an initial query with a WebSocket subscription (AppSync). It delivers an initial snapshot and then pushes incremental updates as mutations occur. The initial snapshot uses an **eventually consistent read**. Subscription events are pushed in near-real-time after writes.

**Where it is used:**

| Model | Location | Filter |
|---|---|---|
| `Game` | `useGameSubscriptions` | `id = gameId` |
| `LineupAssignment` | `useGameSubscriptions` via `useAmplifyQuery` | `gameId` |
| `PlayTimeRecord` | `useGameSubscriptions` via `useAmplifyQuery` | `gameId` |
| `Goal` | `useGameSubscriptions` via `useAmplifyQuery` | `gameId` |
| `GameNote` | `useGameSubscriptions` via `useAmplifyQuery` | `gameId` |
| `PlayerAvailability` | `useGameSubscriptions` via `useAmplifyQuery` | `gameId` |
| `TeamRoster` | `useTeamData` | `teamId` |
| `Player` | `useTeamData` | None (client-side filtered) |
| `FormationPosition` | `useTeamData` | `formationId` |
| `GamePlan` | `useGameSubscriptions` | `gameId` |
| `PlannedRotation` | `useGameSubscriptions` | None (client-side filtered) |

**Key observation:** `PlannedRotation` and `Player` are subscribed **without server-side filters**, meaning the subscription receives all records the user has access to and filters client-side. This is a known workaround for AppSync's 10-filter subscription limit.

---

### 2.2 `.list()` — Paginated Point-in-Time Reads

Used in two specific scenarios where `observeQuery` state may be stale:

**`substitutionService.ts` — Active PlayTimeRecord lookup:**  
When a substitution occurs, the service first checks React state. If the active `PlayTimeRecord` is not found (due to eventual consistency lag between write and subscription delivery), it falls back to a paginated `.list()` query with `gameId` filter, looping through pages with `limit: 1000`.

**`substitutionService.ts` — End-of-game record closure:**  
`closeActivePlayTimeRecords` performs a paginated `.list()` scan, waits 500ms, then performs a **second paginated scan** as a retry to catch records missed by the first scan due to eventual consistency. This is a manual compensation pattern for the lack of strong consistency.

**`useGameSubscriptions.ts` — Lineup sync guard:**  
Before syncing a game plan's starting lineup, a `.list()` call checks whether `LineupAssignment` records already exist for the game, preventing duplicate creation on re-render.

**`cascadeDeleteService.ts` — `listAll()` helper:**  
A paginated `listAll()` utility loops through all pages to collect complete record sets before cascade deletion. Used for `PlayTimeRecord`, `Goal`, and `GameNote` impact queries.

---

### 2.3 `.get()` — Single Item Lookup

Used in `UserProfile.tsx` to load `Team` records for pending invitations:

```typescript
const teamResults = await Promise.all(
  teamIds.map((id) => client.models.Team.get({ id }))
);
```

This is a primary key lookup and is the most efficient DynamoDB read pattern. However, it uses eventual consistency by default.

## 3. Consistency Analysis by Use Case

### 3.1 Game Day Operations (High Sensitivity)

Game day is the most consistency-sensitive context. Coaches are making real-time decisions based on lineup and play time data. The following operations are at risk from eventual consistency:

**Substitution race condition (currently mitigated with workaround):**  
When `executeSubstitution` runs, it creates a `PlayTimeRecord` and immediately needs to close it on the next substitution. Because `observeQuery` delivers updates asynchronously, the newly created record may not yet be in React state when the next substitution fires. The current solution is a fallback `.list()` DB query — this works but adds latency and a full table scan on `PlayTimeRecord` filtered only by `gameId` (no GSI).

**End-of-game record closure (currently mitigated with 500ms retry):**  
`closeActivePlayTimeRecords` uses a 500ms sleep + retry scan to catch records missed by eventual consistency. This is a fragile pattern — on slow connections or under load, 500ms may not be sufficient.

**Recommendation:** The Amplify Gen2 GraphQL client does not expose a `consistentRead` option, so the fix must come from application design rather than a frontend read flag. Keep the race-condition mitigation, but make it more deterministic by favoring idempotent writes, optimistic in-memory state, and server-authoritative mutations where the outcome must be exact (see Section 4, Issues 3 and 4).

---

### 3.2 Pre-Game Planning (Low Sensitivity)

Lineup builder, game planner, and availability grid all use `observeQuery`. These are collaborative but not time-critical — a 1-2 second eventual consistency window is acceptable. No changes recommended here.

---

### 3.3 Season Reports (Read-Heavy, Batch)

`SeasonReport.tsx` and related components aggregate `PlayTimeRecord`, `Goal`, and `GameNote` data. These are post-game reads where data is stable. Eventual consistency is fine. No changes recommended.

---

### 3.4 Cascade Deletes (Correctness-Critical)

`cascadeDeleteService.ts` uses server-side Lambda mutations (`deleteGameSafe`, `deleteTeamSafe`, etc.) rather than client-side deletes. This is the correct pattern — Lambda has direct DynamoDB SDK access and can use strongly consistent reads when needed. No client-side changes needed here.

## 4. Identified Issues and Recommendations

### Issue 1: `PlayTimeRecord` Scans Without GSI

**Current behavior:**  
`.list()` calls in `substitutionService.ts` filter by `gameId` using a DynamoDB Scan with a filter expression. The schema defines a GSI for `PlayTimeRecord`:

```typescript
.secondaryIndexes((index) => [
  index('gameId').queryField('listPlayTimeRecordsByGameId'),
])
```

However, the client code uses `.list({ filter: { gameId: { eq: gameId } } })` instead of the GSI query field. This results in a full table scan filtered post-read rather than a targeted GSI Query.

**Recommendation:** Use the GSI query field for all `PlayTimeRecord` lookups by `gameId`:

```typescript
// Instead of:
client.models.PlayTimeRecord.list({ filter: { gameId: { eq: gameId } } })

// Use:
client.models.PlayTimeRecord.listPlayTimeRecordsByGameId({ gameId })
```

**Tradeoff:** GSI queries are eventually consistent. That means this change improves efficiency, but it does not by itself eliminate race windows in substitution fallback. In this stack, the way to close that gap is to combine the GSI query with idempotent write design and local state reconciliation, or move the authoritative close-out logic to a server-side Lambda/custom resolver when exact read-after-write behavior is required.

**Cost impact:** GSI queries consume fewer RCUs than scans on large tables. For a typical game with 20-50 play time records, the difference is small today but will grow with data volume.

---

### Issue 2: Unfiltered `PlannedRotation` and `Player` Subscriptions

**Current behavior:**  
`PlannedRotation` is subscribed without a server-side filter in `useGameSubscriptions`. `Player` is subscribed without a filter in `useTeamData`. Both are filtered client-side after delivery.

**Why it exists:**  
AppSync WebSocket subscriptions support a maximum of 10 filter conditions. Teams with more than 10 players cannot use a server-side `OR` filter across player IDs.

**Recommendation (low priority):**  
For `PlannedRotation`, add a `gamePlanId` filter once the game plan ID is known. This is a single equality filter and well within the 10-filter limit. The current code already has `currentGamePlanId` available:

```typescript
client.models.PlannedRotation.observeQuery({
  filter: { gamePlanId: { eq: currentGamePlanId } },
})
```

For `Player`, the current approach is acceptable given the AppSync filter limit constraint. Document this as a known limitation.

**Cost impact:** Unfiltered subscriptions deliver all accessible records to the client on initial sync. For coaches with large player pools across many teams, this increases data transfer and RCU consumption. Low risk today; revisit if player pool grows beyond ~200 players per coach.

---

### Issue 3: 500ms Retry in `closeActivePlayTimeRecords`

**Current behavior:**  
After closing active play time records at end-of-game, the code waits 500ms and re-scans to catch records missed by eventual consistency.

**Recommendation:**
The Amplify Gen2 client does not support `consistentRead`, so the retry loop should be treated as a temporary compensation pattern rather than the target design. The practical alternatives are:

- **Track newly created records in memory** within `executeSubstitution` and merge them into the records passed to `closeActivePlayTimeRecords`, reducing dependence on a follow-up read.
- **Make close-out idempotent** so repeated attempts safely converge on the same final record state if a retry is still needed.
- **Custom Lambda resolver** — implement `closeActivePlayTimeRecords` as a server-side mutation using DynamoDB SDK reads plus conditional/atomic write logic. This is the most authoritative option when end-of-game closure must be exact.

The in-memory merge plus idempotent write approach is the best balance of correctness and simplicity on the current stack. A longer retry delay is only a stopgap; it may reduce failures but does not remove the underlying race.

---

### Issue 4: `LineupAssignment` Sync Guard Uses Eventual Read

**Current behavior:**  
In `useGameSubscriptions`, before syncing a game plan's starting lineup, the code queries:

```typescript
const existingAssignments = await client.models.LineupAssignment.list({
  filter: { gameId: { eq: game.id } },
});
```

If this read returns stale data (empty) due to eventual consistency, duplicate lineup assignments will be created.

**Recommendation:**
`consistentRead` is not available through the Amplify Gen2 client. The better fix is to make duplicate creation impossible or harmless:

- Keep `lineupSyncInProgressRef` as the first-line client guard.
- Continue checking local `lineup` state before querying the backend.
- Make the sync write path idempotent, for example by using deterministic assignment identities or a server-side mutation that conditionally creates missing assignments only.
- If duplicate prevention must be authoritative across devices, move lineup sync into a Lambda/custom resolver that performs the check-and-create logic atomically on the server.

The DB query should be treated as a best-effort safety net, not as the sole correctness mechanism.

## 5. Consistency Decision Framework for This Stack

Use this framework when deciding how to handle eventual consistency in Amplify Gen2:

| Criteria | Prefer Client Eventual Read | Prefer Idempotent/Merged Client Design | Prefer Server-Authoritative Lambda or Custom Resolver |
|---|---|---|---|
| Data freshness requirement | Seconds of lag acceptable | Recent writes may be missing briefly, but client can safely merge local knowledge | Must reflect the latest committed state before proceeding |
| Operation frequency | High frequency, UI-driven reads | Per-substitution or per-transition workflows | Lower-frequency but correctness-critical transitions |
| Failure mode | Stale UI that self-corrects | Temporary read gaps without duplicate or conflicting writes | Duplicate records, missed close-out, incorrect final game state |
| Cross-device coordination | Not required | Limited; local tab/session can reconcile its own writes | Required across coaches/devices |
| Implementation cost | Lowest | Moderate | Highest |

**Rule of thumb for TeamTrack:**
- `observeQuery` subscriptions: keep eventual reads on the client; this is the correct default.
- Fallback queries triggered by missing data: use them as best-effort diagnostics or recovery, not as the sole correctness mechanism.
- Write flows that must tolerate stale reads: prefer idempotent mutations and local in-memory merge of freshly written state.
- Write flows that must be authoritative across devices: use Lambda/custom resolver logic with conditional or atomic server-side behavior.
- Report/analytics reads: keep eventual reads; the data is stable post-game.

## 6. Summary of Recommended Changes

| Location | Change | Priority | Cost Impact |
|---|---|---|---|
| `substitutionService.ts` — fallback `.list()` | Already correct pattern; `consistentRead` not available via Amplify client | N/A | — |
| `substitutionService.ts` — end-of-game retry | Merge newly created records in-memory and make close-out idempotent instead of relying on retry timing | Medium | Eliminates or reduces extra scan |
| `useGameSubscriptions.ts` — lineup sync guard | Rely on `lineupSyncInProgressRef` + local state fast path, and make lineup sync idempotent or server-authoritative if duplicates must be impossible | Low | — |
| `substitutionService.ts` — PlayTimeRecord lookup | Use `listPlayTimeRecordsByGameId` GSI query field | Medium | Reduces RCU on large datasets |
| `useGameSubscriptions.ts` — PlannedRotation subscription | Add `gamePlanId` filter | Low | Reduces subscription data transfer |

---

## 7. What Not to Change

- **`observeQuery` subscriptions** — These are correctly using eventual consistency. The subscription mechanism self-heals stale state within milliseconds. Adding strong consistency here would double read costs with no practical benefit.
- **Season report reads** — Post-game data is stable. Eventual consistency is correct.
- **Cascade delete mutations** — Already handled server-side via Lambda. No client changes needed.
- **`.get()` calls in `UserProfile`** — Team data for invitation display is not time-critical. Eventual consistency is fine.
